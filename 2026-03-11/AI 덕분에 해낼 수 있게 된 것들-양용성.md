- 전문 분야 : 데이터베이스 관리 도구 개발, 델파이 애플리케이션 아키텍처 설계

## AI 덕분에 해낼 수 있게 된 것들

### ODAC, MyDAC, SDAC, IBDAC, FireDAC 등 다양한 데이터 액세스 컴포넌트를 통합한 SQLGateDAC 개발
- AI를 활용하여 다양한 데이터베이스와의 호환성을 높이고, 개발 효율성을 극대화
- SQLGateDAC는 SQLGate 제품의 핵심 컴포넌트로, 다양한 데이터베이스에 대한 액세스를 단일 인터페이스로 제공하게 됨

### 도커를 이용한 통합 테스트 데이터베이스 구축
- 배경
  - 기존에는 테스트 데이터베이스 구축이 복잡하고 시간이 많이 소요되었음.
  - 노트북에 테스트 데이터베이스를 구축하는 과정에서 발생하는 문제점들 (예: 환경 설정, 데이터 일관성 등)
  - 노트북 리소스를 너무 많이 사용하여 개발 환경이 불안정해지는 문제. CPU, 메모리 사용량 증가로 인해 데이터베이스를 껏다 켰다 해야 하는 번거로움
  - AI를 활용하여 도커 기반의 통합 테스트 데이터베이스 구축 방법을 개발

- 활용 방법
  - 도커를 이용하여 테스트 데이터베이스를 컨테이너로 구축, 관리하는 방법을 AI가 제안
  - AI의 도움으로 도커 이미지 생성, 컨테이너 관리, 데이터베이스 초기화 스크립트 작성 등의 작업을 자동화
  - 결과적으로 테스트 데이터베이스 구축 시간이 대폭 단축되고, 개발 환경의 안정성이 향상됨

- 결과물 
  - Oracle :  11g, 18c, 21c, 23c, 26g
  - Oracle Wallet :  21g
  - MySQL : 5.7, 8.0, 8.4, 9.6, 
  - MySQL - SSL : 5.7, 8.0, 8.4, 9.6
  - PostgreSQL : 12, 13, 14, 15, 16, 17,  18
  - PostgreSQL - SSL : 12, 13, 14, 15, 16, 17,  18
  - SQL Server : 2017, 2019, 2022, 2025
  - SQL Server - SSL : 2017, 2019, 2022, 2025  
  - 추후 Tibero, Cubrid 등 국내 데이터베이스도 추가 예정
  - 도커 웹서비스 : Portainer

## SQLGateDAC 과 도커 서버간에 테스트 자동화 구축
- SQLGateDAC 을 개발하고 테스트하는 과정에서, 도커 서버와의 통합 테스트 자동화 구축이 필요하다는 것을 인식
- AI를 활용하여 SQLGateDAC과 도커 서버 간의 통합 테스트 자동화 구축 방법을 개발
- AI의 도움으로 테스트 시나리오 작성, 테스트 스크립트 생성, 테스트 실행 및 결과 분석 등의 작업을 자동화
- 결과적으로 SQLGateDAC의 품질과 안정성이 향상되고, 개발 효율성이 극대화됨

## AI를 활용한 델파이 포메터 개발
- AI를 활용하여 델파이 코드의 일관성을 유지하고, 중복된 코드를 줄이는 방향으로 아키텍처를 개선하기 위해 델파이 포메터 개발
- AI의 도움으로 코드 스타일 가이드라인을 정의하고, 이를 기반으로 코드를 자동으로 포맷팅하는 도구를 개발
- 결과적으로 코드의 품질과 유지보수성이 향상되고, AI의 토큰 사용량도 줄어들어 효율적인 코드 생성을 가능하게 됨

## AI를 활용하게 된 배경
SQLGateDAC 개발은 SQLGate for Oracle,MySQL, PostgreSQL, SQL Server 등 다양한 데이터베이스에 대한 액세스를 단일 인터페이스로 제공하는 것을 목표로 했습니다. 이를 위해서는 각 데이터베이스의 특성과 API를 깊이 이해하고, 이를 통합하는 아키텍처를 설계해야 했습니다. AI는 이러한 복잡한 작업을 자동화하고 최적화하는 데 큰 도움을 주었습니다.

사실 각 제품들은 기존의 코드를 복사해서 붙여넣는 방식으로 개발이 진행되어 왔습니다. 하지만 AI를 활용하여 코드의 중복을 줄이고, 공통된 로직을 추출하여 재사용할 수 있는 구조로 리팩토링 하고 싶었습니다.

하지만 동일한 기능의 다른 이름의 컴포넌트와 재사용할 수 있지만 클래스 이름이 다른 컴포넌트가 많아서, AI가 이를 인식하고 적절히 통합하는 데 어려움이 있었습니다. 예를 들어, ODAC의 TOracleConnection과 MyDAC의 TMySQLConnection은 기능적으로 유사하지만 클래스 이름이 다르기 때문에 AI가 이를 동일한 인터페이스로 통합하는 과정에서 혼란을 겪었습니다.

결국 AI를 활용한 델파이 포메터를 개발하여, 코드의 일관성을 유지하고, 중복된 코드를 줄이는 방향으로 아키텍처를 개선할 수 있었습니다. 
만약 포메터를 개발하지 않았다면 AI 가 코드를 일관되게 작성하는 데 어려움을 겪었을 것이며, 결과적으로 코드의 품질과 유지보수성이 저하되었을 것입니다.
또한 AI의 토큰 사용량이 많아지는 문제도 있었는데, 포메터를 개발하여 코드의 일관성을 유지함으로써 AI가 더 효율적으로 코드를 생성할 수 있도록 했습니다.

## 아키텍처 향상에 대한 고민
SQLGateDAC 개발 과정에서 AI를 활용하여 아키텍처를 개선하는 데 많은 고민이 있었습니다. 특히, 다양한 데이터베이스에 대한 액세스를 단일 인터페이스로 제공하는 과정에서 발생하는 코드 중복과 유지보수성 문제를 해결하는 것이 큰 도전이었습니다.

또한 AI 와 지속적으로 탐색하고, 다양한 아키텍쳐를 시도해보면서, 최적의 아키텍처를 찾는 과정에서 많은 시행착오가 있었습니다. AI가 제안하는 다양한 아키텍처를 평가하고, 실제로 적용해보면서, 가장 효율적이고 유지보수하기 쉬운 구조를 찾아내는 데 많은 노력이 필요했습니다.

AI가 제안하는 아키텍처는 때로는 너무 복잡하거나, 실제 개발 환경에 맞지 않는 경우도 있었기 때문에, 이를 적절히 조정하고 최적화하는 과정에서 많은 고민이 있었습니다.

AI의 제안을 무조건적으로 수용하기보다는, 실제 개발 환경과 팀의 역량을 고려하여 최적의 아키텍처를 선택하는 것이 중요하다고 생각했습니다.
